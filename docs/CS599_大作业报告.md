# CS599 大作业报告：智能旅行助手 (AI Travel Agent)

> **方向一：Agentic AI 原生开发**
>
> 基于 Multi-Agent 架构的智能旅行规划系统，集成高德地图 MCP 服务，支持 CLI 和 Web 双界面。

---

## 一、选题背景与设计思想

### 1.1 问题定义

传统旅行规划面临三大痛点：

| 痛点 | 表现 |
|------|------|
| **信息碎片化** | 天气查一个 APP，酒店看另一个，景点攻略散落各处，用户需在多个平台反复切换 |
| **缺乏个性化编排** | 通用攻略无法根据"家庭出游"、"历史文化深度游"等偏好自动调整行程顺序和时间分配 |
| **无预算联动** | 景点门票、酒店、餐饮费用各自独立，难以实时汇总整体花费 |

**核心命题**：能否构建一个 AI 系统，用户只需用自然语言说出"长沙3日游，喜欢自然风光和历史文化，中等预算"，就能自动完成天气查询、景点搜索、酒店推荐、路线规划和预算汇总？

### 1.2 现有方案不足

| 方案 | 局限 |
|------|------|
| 传统旅行 APP（携程、马蜂窝） | 搜索为主，无智能编排；需要用户逐一浏览、比较、决策 |
| 通用大模型对话（ChatGPT） | 能生成文本攻略，但无法实时查询真实天气、酒店价格和地理坐标 |
| 单一 Agent 方案 | 缺乏领域分工，提示词过长导致注意力衰减，工具调用混乱 |
| 硬编码工作流 | 灵活度低，无法根据用户需求动态调整任务顺序 |

### 1.3 项目价值

1. **实用性**：解决真实生活场景的旅行规划需求，输出含经纬度坐标、价格区间的可执行方案
2. **学术价值**：验证 Multi-Agent 协作 + MCP 工具集成 + 流式输出在复杂规划任务中的可行性
3. **工程可复用性**：插件化架构使得更换 LLM、替换地图服务、扩展新 Agent 类型非常方便

### 1.4 技术路线

选取 **Agentic AI 原生开发** 方向，核心技术决策：

```
技术路线：LangGraph 多 Agent 编排
    ├── LLM 推理层: 通义千问 (qwen3.7-plus) via OpenAI 兼容 API
    ├── Agent 框架: LangGraph (ReAct 循环 + 流式事件)
    ├── 工具集成: MCP (Model Context Protocol) 连接高德地图
    ├── 界面层: CLI (原生 Python) + Web (Streamlit)
    └── 架构模式: Planner → Specialist → MCP Tools 三层嵌套
```

**为什么选择 MCP 而非直接调用 REST API**：
- MCP 提供标准化的工具发现机制，新增服务无需修改调用代码
- 工具 Schema 自动生成，减少手写工具定义
- 连接管理与认证统一封装

```

    TripPlanner     ──  create_agent() → ReAct 循环
```

- Planner 通过 `@tool` 装饰器将子 Agent 包装为工具，保持接口一致性

## 四、关键实现与代码展示

### 4.1 Agent 核心循环 (planner.py)

TripPlanner 是整个系统的核心，完成四个阶段：

```
Phase 1: 按领域加载 MCP 工具
  poi_tools   = mcp.get_tools_for("poi")
  weather_tools = mcp.get_tools_for("weather")
  route_tools = mcp.get_tools_for("route")

Phase 2: 创建 3 个 SpecialistAgent
  HotelAgent(llm, "HotelAgent", HOTEL_AGENT_PROMPT, poi_tools)
  AttractionAgent(llm, "AttractionAgent", ATTRACTION_AGENT_PROMPT, poi_tools)
  WeatherAgent(llm, "WeatherAgent", WEATHER_AGENT_PROMPT, weather_tools)

Phase 3: 将子 Agent 包装为 @tool 函数
  search_hotel    → self._hotel_agent.invoke(query)
  search_attraction → self._attraction_agent.invoke(query)
  query_weather   → self._weather_agent.invoke(query)

Phase 4: 创建总控 Agent
  create_agent(llm, [search_hotel, search_attraction, query_weather, *route_tools],
               PLANNER_AGENT_PROMPT)
```

**子 Agent 作为 Tool 的设计**（核心架构决策）：

```python
# 将 SpecialistAgent 的 invoke() 封装为 LangChain Tool
@tool
async def search_hotel(query: str) -> str:
    """搜索酒店。输入城市+偏好，返回酒店列表。"""
    return await self._hotel_agent.invoke(query)
```

Planner 看到的就是三个"黑盒工具"——它不需要知道 HotelAgent 内部也是一个完整的 LangGraph Agent。

**流式输出的事件处理**：

```python
async def stream(self, user_input: str) -> AsyncIterator[str]:
    await self.build()
    async for event in self._agent.astream_events(
        {"messages": [{"role": "user", "content": user_input}]},
        version="v2",
    ):
        kind = event.get("event", "")
        if kind == "on_chat_model_stream":
            content = event["data"]["chunk"].content
            if content:
                # 过滤子 Agent 内部 TOOL_CALL 格式泄漏
                content = _TOOL_CALL_PATTERN.sub("", content)
                if content.strip():
                    yield content
        elif kind == "on_tool_start":
            name = event.get("name", "unknown")
            emoji, label = TOOL_LABELS.get(name, ("🔧", name))
            yield f"\n{emoji} {label}...\n"
```

#### MCP 客户端单例 (mcp_client.py)

```python
class McpClientManager:
    _instance = None

    def __new__(cls):
        if cls._instance is None:
            cls._instance = super().__new__(cls)
            cls._instance._initialized = False
        return cls._instance

    async def get_tools_for(self, domain: str) -> list[BaseTool]:
        all_tools = await self.get_all_tools()       # 首次请求后缓存
        target_names = set(CONFIG.tool_domains.get(domain, []))
        return [t for t in all_tools if t.name in target_names]
```

**设计要点**：
- `__new__` + `_initialized` 双重单例保护，确保全局只有一个 MCP 连接
- 工具按领域（poi / weather / route）分发，每个 SpecialistAgent 只拿到自己需要的工具子集
- 全量工具只请求一次并缓存，后续从缓存过滤

### 4.3 配置文件设计 (config.py)

```python
@dataclass
class Config:
    api_key: str = field(default_factory=lambda: os.getenv("DASHSCOPE_API_KEY", ""))
    model_name: str = "qwen3.7-plus"
    temperature: float = 0.7
    base_url: str = "https://dashscope.aliyuncs.com/compatible-mode/v1"

    def create_llm(self) -> ChatOpenAI:
        return ChatOpenAI(
            model=self.model_name,
            api_key=self.api_key,
            base_url=self.base_url,
            temperature=self.temperature,
            streaming=True,
        )
```

**设计亮点**：
- `@dataclass` 定义配置，模块级 `CONFIG` 实例作为单例
- `__post_init__` 自动校验 API Key
- `create_llm()` 工厂方法，隔离 LLM 构造细节
- 通过 OpenAI 兼容接口调用通义千问，模型切换只需改 `model_name` 和 `base_url`

### 4.6 🌟 前沿技术融合：MCP 协议 × Agentic RAG（加分项）

#### 4.6.1 MCP 协议深度集成

本项目是国内较早将 **MCP (Model Context Protocol)** 应用于实际 Multi-Agent 系统的项目之一。MCP 提供了标准化的工具发现与调用机制：

```
传统方式：手写 API 调用
  ├── 每个 API 单独封装函数
  ├── 认证方式各自不同
  └── 工具 Schema 需手动维护

MCP 方式：标准化工具集成
  ├── 统一工具发现 (client.get_tools())
  ├── 自动 Schema 生成
  └── 集中认证管理
```

**MCP 工具按领域分发机制**：

```python
# 15 个 MCP 工具按领域自动分发到不同的 SpecialistAgent
tool_domains = {
    "poi":     ["maps_text_search", "maps_search_detail"],    # → HotelAgent / AttractionAgent
    "weather": ["maps_weather"],                              # → WeatherAgent
    "route":   ["maps_direction_*"],                          # → Planner 直接使用
}
```

每个 SpecialistAgent 只持有自己领域的工具子集 —— 这是 **最小权限原则** 在 AI Agent 系统中的实践。

#### 4.6.2 Agentic RAG 旅行知识库

在传统 RAG（Retrieval-Augmented Generation）基础上，本项目实现了 **Agentic RAG** 模式：Agent 自主决定何时检索、检索什么、如何使用检索结果。

**与传统 RAG 的区别**：

| 传统 RAG | Agentic RAG（本项目） |
|----------|----------------------|
| 被动：每次查询都先检索再生成 | 主动：Agent 判断是否需要检索 |

#### 4.7.2 容错与降级

**MCP 连接重试机制**（指数退避）：

```python
MAX_RETRIES = 3       # 最大重试次数
RETRY_DELAY = 2.0     # 基础延迟（秒）
# 重试间隔: 2s → 4s → 6s（指数增长）
```

**熔断保护**：
- 连续 3 次失败 → 自动熔断，拒绝后续请求
- MCP 返回 429（限流）→ 立即熔断

**降级策略**：
- MCP 不可用时 → 返回空工具列表（不崩溃）
- LLM 通过自身知识 + RAG 知识库补充缺失数据
- Web 端显示降级提示："部分服务暂时不可用，AI 已使用本地知识生成计划"

### 5.2 Agent 行为评估

#### 工具调用链路追踪

```
[🌤️ 查询天气] → maps_weather 调用 → 返回当前4天预报
[🏨 搜索酒店] → maps_text_search 调用 → 返回五一广场酒店列表
[🏛️ 搜索景点] → maps_text_search 调用 (×2) → 返回自然/文化景点
[🔧 maps_weather] → Planner 直接调用路线工具
```

**观察**：Agent 按照 Prompt 定义的五阶段工作流顺序执行，未出现跳过阶段或重复调用。

#### 降级行为

当 MCP 工具因配额限制返回错误时，LLM 会自动用训练知识补充：
- 酒店信息：基于城市知识给出全季酒店等推荐
- 景点信息：基于训练数据提供岳麓山、橘子洲等真实景点
- 天气信息：无法查历史天气时，返回当前预报并告知用户

## 七、课程总结

### 7.1 个人收获

#### 技术层面

1. **深入理解 Agent 架构**：从零实现了 Planner → Specialist → MCP Tools 三层嵌套 Agent 系统，理解了 ReAct 循环的执行机制和流式事件处理
2. **掌握 MCP 协议**：学习了 Model Context Protocol 的标准工具发现和调用流程，体会到标准化协议对 AI 生态的价值
3. **工程化思维**：单例模式、工厂方法、适配器模式在真实 AI 项目中的落地，不再是教科书上的抽象概念
4. **全栈 AI 开发**：从 LLM 调用、Agent 编排到 CLI/Web 双界面，覆盖了 AI 应用的完整开发链路

#### 工程思维转变

| 之前 | 之后 |
|------|------|
| "用哪个模型好用就行" | 模型是插件，架构设计决定系统的扩展性和可维护性 |
| "Agent 就是一个 prompt" | Agent = System Prompt + Tools + ReAct Loop + Event Stream，缺一不可 |
| "工具调用就是调 API" | MCP 将工具标准化，从紧耦合变为可插拔 |
| "界面是后加的事" | CLI 和 Web 共享同一套核心逻辑，界面层只做渲染 |

#### 踩过的坑

1. **ChatTongyi 流式 Bug**：`langchain_community` 缺少 Key 存在性检查导致 `KeyError`——框架 bug 需自行修复
2. **编码问题**：Windows GBK vs UTF-8——跨平台开发的经典问题
3. **MCP 配额**：免费额度是真实限制——Demo 和产品之间有鸿沟
4. **RAG 关键词匹配**：中文分词在无 jieba 依赖下需用 2-gram 近似——工程折衷的典型案例
5. **Streamlit Cloud 路径**：`sys.path` 需显式添加项目根目录——理解 Python 模块解析机制

### 7.2 对课程的建议

1. **增加 MCP 实践课时**：MCP 是 2025 年最重要的 AI 协议之一，值得更深入讲解
2. **提供统一的 API 配额**：学生自费购买 API 额度成本较高，建议课程提供统一额度
3. **Agent 调试工具**：Agent 的执行链路长，调试困难，建议介绍 LangSmith 等可观测性工具
4. **增加代码 Review 环节**：同学互审 Agent 架构设计，比单纯提交代码更有教学价值
