# CS599 大作业报告：智能旅行助手 (AI Travel Agent)

> **方向一：Agentic AI 原生开发**
>
> 基于 Multi-Agent 架构的智能旅行规划系统，集成高德地图 MCP 服务，
> 融合 Agentic RAG 检索增强生成，支持 CLI 和 Web 双界面，
> 部署于 Streamlit Cloud 提供公网访问。

---

## 摘要

本报告详细介绍了一个基于 **Multi-Agent 架构** 的智能旅行规划系统的设计与实现。该系统采用 LangGraph 作为 Agent 编排框架，通过 MCP（Model Context Protocol）协议集成高德地图 15 个实时服务工具，创新性地融合了 Agentic RAG 旅行知识检索机制，实现了从用户自然语言需求到完整旅行方案的自动化生成。系统架构遵循"Planner → Specialist → MCP Tools"三层嵌套模式，每个 SpecialistAgent 遵循最小权限原则仅持有领域相关工具。在工程实践方面，系统实现了结构化日志追踪、MCP 连接容错熔断、输入校验与安全防护等生产级特性，并通过 Streamlit Cloud 完成云部署提供公网访问。经过 10 项功能测试验证，系统能够稳定输出包含天气、酒店、景点、餐饮、交通和预算的结构化旅行计划。

**关键词**：Multi-Agent 系统、MCP 协议、Agentic RAG、LangGraph、大语言模型、旅行规划

## 一、选题背景与设计思想

### 1.1 问题定义

#### 1.1.1 旅行规划的复杂性

旅行规划本质上是一个**多约束优化问题**，涉及信息获取、偏好匹配、时间编排、预算控制等多个维度。传统模式下，用户需要自行完成以下步骤：

1. **信息收集阶段**：在多个 APP 和网站间切换，查询目的地天气、搜索景点信息、比较酒店价格、了解交通方式
2. **方案编排阶段**：根据个人偏好和时间约束，将零散信息组合成有序的每日行程
3. **预算核算阶段**：汇总景点门票、住宿、餐饮、交通等各项费用

这一过程的痛点集中体现在三个方面：

| 痛点 | 表现 | 影响 |
|------|------|------|
| **信息碎片化** | 天气查天气 APP，酒店看携程/飞猪，景点攻略散落在小红书/马蜂窝/知乎，用户需在 5+ 个平台反复切换 | 信息整合耗时长（通常 2-4 小时），容易遗漏关键信息 |
| **缺乏个性化编排** | 通用攻略无法根据"家庭出游带老人"、"历史文化深度游"、"蜜月休闲度假"等偏好自动调整行程节奏和时间分配 | 用户需手动筛选和调整，试错成本高 |
| **无预算联动** | 景点门票、酒店、餐饮、交通费用各自独立计算，缺乏实时汇总和超标预警 | 经常出现"玩得开心但超预算"的情况 |

#### 1.1.2 核心命题

> **能否构建一个 AI 系统，用户只需用自然语言说出"长沙3日游，喜欢自然风光和历史文化，中等预算，住五一广场"，就能自动完成天气查询、景点搜索、酒店推荐、路线规划和预算汇总？**

这一命题触及了 AI Agent 研究的核心问题：**如何让 AI 自主使用外部工具完成复杂的、多步骤的现实任务？**

### 1.2 现有方案不足

#### 1.2.1 方案对比分析

| 方案类别 | 代表产品 | 优势 | 局限 | 与本项目差距 |
|----------|---------|------|------|-------------|
| 传统旅行 APP | 携程、马蜂窝、去哪儿 | 数据全面、可在线预订 | 搜索为主，无智能编排；用户需逐一浏览、比较、决策 | 缺失 AI 编排能力 |
| 通用大模型对话 | ChatGPT、Kimi、豆包 | 能生成文本攻略，回答旅行问题 | 无法实时查询真实天气、酒店价格和地理坐标；信息可能过时或幻觉 | 缺失实时工具调用 |
| 单一 Agent 方案 | AutoGPT、MetaGPT | 能自主分解任务 | 缺乏领域分工，提示词过长导致注意力衰减，工具调用混乱 | 缺失领域专业化分工 |
| 硬编码工作流 | 传统后端编排（Airflow、Temporal） | 稳定可靠，可监控 | 灵活度低，无法根据用户需求动态调整任务顺序和策略 | 缺失 AI 自主决策 |
| RAG 增强方案 | 结合向量库的问答系统 | 能检索最新知识 | 检索与生成分离，缺乏多步推理和工具调用能力 | 缺失 Agentic 主动决策 |

#### 1.2.2 技术空白分析

当前市场上存在一个明显的技术空白：**缺乏一个将 Multi-Agent 协作、实时工具调用（MCP）、知识检索增强（RAG）和用户友好界面有机结合的旅行规划系统**。本项目正是瞄准这一空白展开。

### 1.3 项目价值

#### 1.3.1 实用价值

- **效率提升**：将传统 2-4 小时的旅行规划缩短至 1-2 分钟
- **决策质量**：AI 综合天气、地理位置、开放时间等多维因素，生成逻辑合理的行程
- **可执行性**：输出含经纬度坐标、价格区间、预约方式的具体方案，非空泛建议

#### 1.3.2 学术价值

- **Multi-Agent 协作模式验证**：验证了"Planner + Specialist"双层架构在复杂规划任务中的有效性
- **MCP 协议实践**：为 MCP 在 Multi-Agent 系统中的应用提供了工程参考
- **Agentic RAG 创新**：提出了"Agent 自主决定检索时机"的 RAG 使用新模式

#### 1.3.3 工程可复用性

- **插件化 LLM**：切换模型只需修改 3 行配置，已验证 Tongyi → OpenAI 兼容的无缝切换
- **可拔插工具层**：MCP 标准化接口使地图服务可替换为其他 MCP Server
- **模块化 Agent**：新增领域 Agent 只需 2 个文件（Agent 定义 + Prompt），自动注册为 Planner 工具

### 1.4 技术路线

#### 1.4.1 为什么选择 Agentic AI 原生开发方向

本课程提供了两个方向：方向一为"Agentic AI 原生开发"，方向二为"企业级应用软件的 Agent 改造"。本项目选择方向一，核心理由：

1. **问题本质**：旅行规划是需要自主决策、工具使用和多步推理的复杂任务，天然适合 Agent 架构
2. **技术前沿性**：MCP 协议 2024 年底发布，2025 年成为 AI 工具集成的事实标准，是值得深入探索的前沿方向
3. **无历史包袱**：从零构建可以自由选择最佳实践，不受现有系统约束

#### 1.4.2 核心技术决策

```
技术路线：LangGraph 多 Agent 编排
    │
    ├── LLM 推理层
    │   ├── 模型：通义千问 (qwen3.7-plus)
    │   ├── 接口：OpenAI 兼容 API（ChatOpenAI）
    │   ├── 端点：阿里百炼 DashScope
    │   └── 特性：流式输出 (streaming=True)
    │
    ├── Agent 框架
    │   ├── LangGraph：ReAct 循环 + 流式事件 (astream_events v2)
    │   ├── LangChain：统一的 Agent 创建接口 (create_agent)
    │   └── langchain-mcp-adapters：MCP → LangChain Tool 转换
    │
    ├── 工具集成
    │   ├── MCP (Model Context Protocol)：连接高德地图 15 个工具
    │   ├── Agentic RAG：30+ 条旅行知识自动检索
    │   └── 自定义工具：@tool 装饰器包装子 Agent
    │
    ├── 界面层
    │   ├── CLI：原生 Python + asyncio + 逐 token 流式打印
    │   └── Web：Streamlit + session_state + 响应式布局
    │
    └── 架构模式
        └── Planner → Specialist → MCP Tools 三层嵌套
```

#### 1.4.3 为什么选择 MCP 而非直接调用 REST API

| 维度 | REST API 直接调用 | MCP 协议 |
|------|-------------------|----------|
| 工具发现 | 手动阅读文档，硬编码 URL 和参数 | `client.get_tools()` 自动发现所有可用工具 |
| Schema 管理 | 手动编写 JSON Schema / Pydantic 模型 | 工具 Schema 自动从 MCP Server 获取 |
| 认证管理 | 每个 API 单独处理认证方式 | 通过 MCP Client 统一配置 Authorization Header |
| 新增工具 | 修改代码 + 重新部署 | MCP Server 更新后，Client 自动感知 |
| 多服务管理 | 分别管理多个 API 的 URL、Key、参数 | `MultiServerMCPClient` 统一管理 |
| 生态兼容 | 与特定 API 绑定 | 任何实现 MCP 协议的服务均可接入 |

#### 1.4.4 为什么选择 LangGraph 而非其他 Agent 框架

| 框架 | 优势 | 劣势 | 本项目为什么选它 |
|------|------|------|-----------------|
| **LangGraph** | 原生流式事件 (astream_events v2)、灵活的状态图、与 LangChain 深度集成 | 学习曲线较陡 | ✅ 流式输出是核心需求 |
| CrewAI | 简单易用、多 Agent 角色扮演 | 流式支持有限、定制化能力弱 | ❌ 无法满足逐 token 流式需求 |
| AutoGen | 微软出品、对话式多 Agent | 偏聊天场景、规划类任务支持弱 | ❌ 不适合结构化输出场景 |
| 自建 ReAct | 完全可控 | 工作量大、需自行处理工具调用格式 | ❌ 开发效率低 |

#### 2.2.2 关键设计约束

| 约束 | 描述 | 实现方式 |
|------|------|---------|
| 最小权限原则 | 每个 SpecialistAgent 只能访问自己领域的 MCP 工具 | `tool_domains` 字典 + `get_tools_for(domain)` 过滤 |
| 单例 MCP 连接 | 全局只有一个 MCP 客户端实例，避免重复连接 | `__new__` + `_initialized` 双重单例 |
| 工具缓存 | 全量工具列表仅请求一次，后续从内存缓存读取 | `_tools_cache` 字典缓存 |
| 接口一致性 | 子 Agent 和 MCP 工具对外提供统一接口 | `@tool` 装饰器 + `BaseTool` 抽象 |
| 关注点分离 | 界面层只做渲染，不做业务逻辑 | `render.py` 独立模块，CLI/Web 共享 |

### 4.1 Agent 核心循环 (planner.py — 162 行)

TripPlanner 是整个系统的核心编排器，负责将子 Agent 包装为工具并创建顶层 Planner Agent。

#### 4.1.1 build() 方法的四个阶段

```python
async def build(self):
    """初始化所有子 Agent + 组装 Planner"""
    if self._agent is not None:
        return  # 幂等：已构建则跳过

    # ===== Phase 1: 按领域加载 MCP 工具 =====
    # 策略模式：tool_domains 字典定义工具→领域映射
    poi_tools = await self.mcp.get_tools_for("poi")       # maps_text_search, maps_search_detail
    weather_tools = await self.mcp.get_tools_for("weather") # maps_weather
    route_tools = await self.mcp.get_tools_for("route")    # maps_direction_*_by_address

    # ===== Phase 2: 创建 3 个 SpecialistAgent =====
    # 每个子 Agent 是独立的 LangGraph ReAct Agent
    self._hotel_agent = SpecialistAgent(
        self.llm, "HotelAgent", HOTEL_AGENT_PROMPT, poi_tools
    )
    self._attraction_agent = SpecialistAgent(
        self.llm, "AttractionAgent", ATTRACTION_AGENT_PROMPT, poi_tools
    )
    self._weather_agent = SpecialistAgent(
        self.llm, "WeatherAgent", WEATHER_AGENT_PROMPT, weather_tools
    )
    # 并行构建三个子 Agent
    await self._hotel_agent.build()
    await self._attraction_agent.build()
    await self._weather_agent.build()

    # ===== Phase 3: 将子 Agent 包装为 @tool =====
    # 这是本项目的核心架构决策：
    # Planner 看到的是无状态的工具函数，而非复杂的 Agent 对象
    @tool
    async def search_hotel(query: str) -> str:
        """搜索酒店。输入城市+偏好，返回酒店列表。"""
        async with self._tracer.span("search_hotel", query=query[:50]):
            return await self._hotel_agent.invoke(query)

    @tool
    async def search_attraction(query: str) -> str:
        """搜索景点。输入城市+类型偏好，返回景点列表。"""
        async with self._tracer.span("search_attraction", query=query[:50]):
            return await self._attraction_agent.invoke(query)

    @tool
    async def query_weather(query: str) -> str:
        """查询天气。输入城市+日期，返回天气概况。"""
        async with self._tracer.span("query_weather", query=query[:50]):
            return await self._weather_agent.invoke(query)

    @tool
    async def search_travel_tips(query: str) -> str:
        """检索旅行知识库。输入查询问题，返回相关的旅行贴士。"""
        async with self._tracer.span("RAG检索", query=query[:50]):
            return search_travel_knowledge(query, city="", top_k=5)

    # ===== Phase 4: 创建总控 Agent =====
    all_tools = [
        search_hotel, search_attraction, query_weather,
        search_travel_tips,  # ← Agentic RAG 工具
        *route_tools,        # ← MCP 路线工具（展开）
    ]

    self._agent = create_agent(
        model=self.llm,
        tools=all_tools,
        system_prompt=PLANNER_AGENT_PROMPT,
    )
```

#### 4.1.2 stream() 流式输出 — 事件驱动架构

```python
async def stream(self, user_input: str) -> AsyncIterator[str]:
    """流式输出旅行计划。"""
    await self.build()

    async for event in self._agent.astream_events(
        {"messages": [{"role": "user", "content": user_input}]},
        version="v2",  # LangGraph 事件 v2 API
    ):
        kind = event.get("event", "")

        # ===== 事件类型 1: LLM 流式 Token =====
        if kind == "on_chat_model_stream":
            content = event["data"]["chunk"].content
            if content:
                # 关键 Bug 修复：过滤子 Agent 内部的 TOOL_CALL 文本泄漏
                content = _TOOL_CALL_PATTERN.sub("", content)
                if content.strip():
                    yield content

        # ===== 事件类型 2: 工具调用开始 =====
        elif kind == "on_tool_start":
            name = event.get("name", "unknown")
            emoji, label = TOOL_LABELS.get(name, ("🔧", name))
            yield f"\n{emoji} {label}...\n"

        # ===== 事件类型 3: 工具调用结束 =====
        elif kind == "on_tool_end":
            # 静默处理，避免输出刷屏
            pass
```

#### 4.1.3 `[TOOL_CALL:...]` 泄漏问题与修复

子 Agent 的 System Prompt 要求输出固定格式 `[TOOL_CALL:amap_maps_xxx:...]`，这在子 Agent 内部的 ReAct 循环中是必要的。但当 Planner 通过 `ainvoke()` 获取结果时，这些内部文本可能泄漏到最终输出中。修复方案是在 stream() 中用正则过滤：

```python
_TOOL_CALL_PATTERN = re.compile(r"\[TOOL_CALL:[^\]]*\]")
content = _TOOL_CALL_PATTERN.sub("", content)  # 移除所有 TOOL_CALL 文本
```

### 4.2 领域专家 Agent (specialist.py — 65 行)

SpecialistAgent 封装了单一职责的 ReAct Agent，遵循最小权限原则：

```python
class SpecialistAgent:
    """领域专家智能体。每个实例持有自己领域的 MCP 工具。"""

    def __init__(
        self,
        llm: BaseChatModel,       # 共享的 LLM 实例
        name: str,                # Agent 标识名
        system_prompt: str,       # 专属系统提示词（约束行为）
        tools: list[BaseTool],    # 领域工具子集（最小权限）
    ):
        self.llm = llm
        self.name = name
        self.system_prompt = system_prompt
        self.tools = tools
        self._agent = None        # 懒加载的 LangGraph Agent

    async def build(self):
        """构建底层 LangGraph Agent（懒加载）"""
        if self._agent is None:
            self._agent = create_agent(
                model=self.llm,
                tools=self.tools,
                system_prompt=self.system_prompt,
            )

    async def invoke(self, user_input: str) -> str:
        """非流式调用 — Planner 通过此接口获取子 Agent 结果"""
        await self.build()
        result = await self._agent.ainvoke({
            "messages": [{"role": "user", "content": user_input}]
        })
        # 返回最后一条 AI 消息的文本内容
        return result["messages"][-1].content
```

**设计意图**：
- HotelAgent 和 AttractionAgent 共享 `poi_tools`，但拥有不同的 System Prompt
- 这种"同工具不同提示词"的设计使得 Agent 行为可精确控制，无需修改工具定义
- 每个 SpecialistAgent 是独立的 LangGraph Agent，有独立的 ReAct 循环

#### 4.3.1 单例模式实现

```python
class McpClientManager:
    _instance: "McpClientManager | None" = None

    def __new__(cls) -> "McpClientManager":
        if cls._instance is None:
            cls._instance = super().__new__(cls)
            cls._instance._initialized = False
        return cls._instance

    def __init__(self):
        if self._initialized:    # ← 双重保护
            return
        self._client = None
        self._tools_cache: dict[str, list[BaseTool]] = {}
        self._consecutive_failures = 0  # ← 熔断计数器
        self._circuit_open = False       # ← 熔断开关
        self._initialized = True
```

**为什么需要双重保护**：
- `__new__` 确保全局只有一个实例（类级别）
- `_initialized` 确保 `__init__` 只执行一次（实例级别）
- 原因是 Python 的 `__new__` 返回已有实例后，`__init__` 仍会被调用

#### 4.3.2 容错重试与熔断

```python
MAX_RETRIES = 3
RETRY_DELAY = 2.0  # 基础延迟（指数增长: 2s → 4s → 6s）

async def get_all_tools(self) -> list[BaseTool]:
    if self._circuit_open:
        raise MCPConnectionError("MCP 服务熔断中，请稍后重试")

    last_error = None
    for attempt in range(1, MAX_RETRIES + 1):
        try:
            client = await self._get_client()
            tools = await asyncio.wait_for(
                client.get_tools(), timeout=30.0
            )
            self._consecutive_failures = 0  # 重置计数器
            return tools
        except asyncio.TimeoutError:
            last_error = MCPConnectionError(f"超时 (第{attempt}次)")
        except Exception as e:
            if "500" in str(e):
                last_error = MCPConnectionError(f"服务内部错误 (第{attempt}次)")
            elif "429" in str(e) or "限制" in str(e):
                self._circuit_open = True  # 限流 → 立即熔断
                raise MCPRateLimitError(f"频率超限，已熔断")

        if attempt < MAX_RETRIES:
            await asyncio.sleep(RETRY_DELAY * attempt)  # 指数退避

    self._consecutive_failures += 1
    if self._consecutive_failures >= 3:
        self._circuit_open = True  # 连续失败 → 熔断
    raise last_error
```

#### 4.8.2 Agentic RAG 旅行知识库

**设计理念**：传统 RAG 是"每次查询都先检索再生成"的被动模式。Agentic RAG 让 Agent 自主决定**何时检索、检索什么、如何使用检索结果**。

```
传统 RAG 流程:
  User Query → [必选] Vector Search → LLM + Context → Answer

Agentic RAG 流程 (本项目):
  User Query → Agent thinks →
    ├─ Need knowledge? → [自主触发] Knowledge Search → Add to context
    ├─ Need tools?    → [自主触发] MCP Tool Call → Add to result
    └─ Ready?         → Synthesize → Answer
```

| 故障场景 | 检测方式 | 处理策略 | 用户感知 |

| 输入不合法 | `TravelRequest.validate()` | 返回所有错误列表 | 逐条显示错误提示 |

## 七、课程总结

### 7.1 个人收获

#### 7.1.1 技术层面

1. **深入理解 Agent 架构**：从零实现了"Planner → Specialist → MCP Tools"三层嵌套 Agent 系统。深刻理解了 ReAct 循环的 Think → Act → Observe → Think 执行机制，以及 LangGraph `astream_events v2` 事件系统的使用。知道了 Agent 不是"写一个 prompt 就完事"，而是 System Prompt + Tools + ReAct Loop + Event Stream 的精密协作。

2. **掌握 MCP 协议**：学习了 Model Context Protocol 的标准化工具发现和调用流程。体会到"标准化协议"对 AI 生态的价值——就像 HTTP 之于 Web，MCP 正在成为 AI Agent 工具集成的事实标准。

3. **工程化思维落地**：单例模式（MCP 连接管理）、工厂方法（LLM 创建）、门面模式（TripPlanner）、装饰器模式（@tool 包装）、策略模式（tool_domains）、适配器模式（render.py）——六种设计模式在真实 AI 项目中的实战应用。

4. **全栈 AI 开发能力**：从 LLM API 调用 → Agent 编排 → MCP 工具集成 → 流式事件处理 → CLI/Web 双界面 → 生产级日志/熔断 → 云部署，覆盖了 AI 应用的完整开发链路。

5. **Agentic RAG 创新**：突破了"每次查询都先检索"的传统 RAG 模式，实现了 Agent 自主决定检索时机的 Agentic RAG，这是 RAG 从"被动增强"到"主动检索"的范式升级。

#### 7.1.2 工程思维转变

| 之前 | 之后 | 转变本质 |
|------|------|---------|
| "用哪个模型好用就行" | 模型是插件，架构设计决定系统的扩展性和可维护性 | 从工具思维到架构思维 |
| "Agent 就是一个 prompt" | Agent = System Prompt + Tools + ReAct Loop + Event Stream | 从单一视角到系统视角 |
| "工具调用就是调 API" | MCP 将工具标准化，从紧耦合变为可插拔 | 从具体到抽象 |
| "界面是后加的事" | CLI 和 Web 共享同一套核心逻辑 | 从线性开发到分层架构 |
| "代码能跑就行" | 日志/追踪/熔断/校验是生产级的基础 | 从 Demo 思维到产品思维 |
| "报错是别人的事" | 自定义异常体系让错误可预期、可恢复 | 从被动到主动防御 |

| 2 | Windows 终端 emoji 乱码 | GBK 编码无法编码 🚀 等字符 | `sys.stdout.reconfigure(encoding='utf-8')` | 跨平台必须先处理编码 |
| 3 | MCP 工具 500 错误 | 阿里百炼免费账户配额限制 | 重试 + 熔断 + 降级 + RAG 补充 | Demo 和产品之间有鸿沟 |
| 4 | Streamlit Cloud `ModuleNotFoundError: src` | `sys.path` 不含项目根目录 | `sys.path.insert(0, str(Path(__file__).parent.parent))` | 理解 Python 模块解析机制 |
| 5 | Python 3.14 dataclass TypeError | 有默认值字段后跟无默认值字段 | 所有字段给默认值，保持 dataclass 顺序 | Python 版本升级可能引入更严格检查 |
| 6 | 中文 RAG 检索精度 | 无 jieba 重依赖，2-gram 精度有限 | BM25 变体 + 城市加权 + 内容匹配 | 工程折衷：轻量化 vs 精度 |

### 7.2 对课程的建议

1. **增加 MCP 实践课时**：MCP 是 2025 年最重要的 AI 协议之一，建议安排 2-4 课时深入讲解 MCP Server/Client 开发
2. **提供统一 API 配额**：学生自费购买额度成本高（本课程约消耗 ¥50-100），建议课程提供统一额度或代理
3. **Agent 可观测性工具**：Agent 执行链路长、调试困难，建议介绍 LangSmith、LangFuse 等可观测性工具
4. **增加架构评审环节**：同学互审 Agent 架构设计（如本项目的三层嵌套是否合理），比单纯提交代码更有教学价值
5. **引入 MCP Marketplace**：展示如何为已有服务创建 MCP Server，让"万物皆可 MCP"的理念深入理解

### C. 设计模式应用清单

| 设计模式 | 应用位置 | 解决的问题 | 代码示例 |
|---------|---------|-----------|---------|
| **单例模式** | `McpClientManager` | 确保全局唯一 MCP 连接，避免重复初始化 | `__new__` + `_initialized` |
| **工厂方法** | `Config.create_llm()` | 隔离 LLM 创建细节，支持运行时切换模型 | `return ChatOpenAI(model=..., ...)` |
| **门面模式** | `TripPlanner` | 隐藏内部多 Agent 复杂性，对外提供简单接口 | `planner.stream(user_input)` |
| **装饰器模式** | `@tool` 包装子 Agent | 将 SpecialistAgent 适配为 LangChain Tool 接口 | `@tool\nasync def search_hotel(...)` |
| **策略模式** | `tool_domains` 字典 | 运行时按领域选择工具子集 | `get_tools_for(domain)` |
| **适配器模式** | `render.py` | 将 JSON 数据适配为 CLI 和 Web 两种视图 | `format_plan_cli()` / `st.markdown()` |
